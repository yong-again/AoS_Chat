# 2026-07-09 수정 이력

### 1. 스피어헤드 능력 카테고리 질문 검색 실패 (Regiment Abilities/Enhancements)

**증상**
"Grundstok Trailblazers의 Regiment Abilities와 Enhancements를 알려줘"에
"해당 유닛이나 레지멘트 오브 리나운을 찾을 수 없습니다"로 응답. 데이터는
spearhead_db에 정상 존재("Grundstok Trailblazers > Regiment Abilities" 등).

**원인 (2중 실패)**
- A. 키워드 추출이 대상 고유명을 버림: 'Regiment Ability\nEnhancement'만
  추출(개행 포함). 이 키워드로 faction_db 검색 시 'Regiment' 어휘가 겹치는
  Regiments of Renown(Drekki's Privateers) 청크가 전 팩션에서 상위 점령.
- B. 라우팅 오류: Regiment Abilities는 스피어헤드 전용 구조라 데이터가
  spearhead_db에만 있는데, 질문에 '스피어헤드' 단어가 없어 기존 안전장치
  미발동 → faction_db로 라우팅. 프로브는 spearhead_db 최근접(불일치)을
  로그로만 남기고 버림.

**수정** (`app.py`)
- ① 추출 프롬프트 보강: "고유명 절대 버리지 말 것" 규칙 + 실패 케이스
  예시 2건 추가, "여러 키워드는 한 줄 공백 구분" 규칙 추가. 코드에서도
  search_query 개행→공백 정규화.
- ② 스피어헤드 고유명 사전 교정: spearhead_db 메타데이터의
  spearhead_name 48종을 시작 시 로드(load_spearhead_names, 캐시).
  질문에 이름이 등장하면 spearhead_db로 결정적 교정(안전장치 3).
  'Regiment Ability' 용어도 스피어헤드 전용 명칭이므로 신호로 추가
  (안전장치 4) — 단 Enhancement는 정규 게임 공통 용어(Heroic Traits,
  Artefacts 등 상위 카테고리)라 단독으로는 신호로 쓰지 않음(사용자
  도메인 지식 교정 반영). 라우터 프롬프트에도 동일 힌트 추가.
- ③ 프로브 크로스 DB 폴백: 기존에 로그로만 쓰던 DB별 유사도 프로브를
  소비 — 선택 DB 결과에 키워드가 없고 최근접 DB가 다르면 그 DB를 추가
  검색해 병합. 오라우팅 전반에 대한 범용 회복 장치.

**검증**
- ① 추출 재현: 실패 질문(영/한) 모두 'Grundstok Trailblazers Regiment
  Abilities Enhancements'로 고유명 보존 + 한 줄 출력. healing/워든 스탯
  회귀 없음.
- ② 사전 매칭 4케이스: 이름 포함 질문 매칭, '레지먼트 어빌리티' 단독
  용어 신호 발동, '인핸스먼트 추천' 단독은 미발동(정규 게임 존중).
- e2e: 교정된 db+키워드로 최종 컨텍스트에 Grundstok Trailblazers 전체
  5청크(Regiment Abilities·Enhancements 포함) 확인.

### 3. 동시 접속(멀티유저) 안전성 보강

**배경**
Cloudflare Tunnel 공유로 여러 명이 동시 접속하는 상황을 점검한 결과
5가지 문제 발견 → 우선순위별 수정.

**수정**
- 프라이버시 (`app.py`): 공유 모드(APP_ACCESS_CODE 설정)에서 사이드바의
  "저장된 세션 목록/불러오기"와 "개발자 로그 뷰어"를 숨김 — 접속자가
  서로의(호스트 포함) 대화·질문 로그를 열람하던 문제 차단. 로컬(코드
  미설정)에서는 기존과 동일.
- 세션 ID 충돌 (`app.py`): 초 단위 타임스탬프에 uuid4 6자리 접미사 추가
  (`_new_session_id`) — 같은 초에 접속한 두 사용자의 기록이 같은 파일에
  섞이던 문제 해소.
- Chroma 자가 복구 범위 축소 (`app.py`): `st.cache_resource.clear()`
  (전 세션의 모델·BM25 인덱스·진행 중 쿼리까지 무효화) 대신, chroma
  클라이언트 캐시만 비우고 공유 collections dict를 제자리 갱신 —
  복구를 트리거한 세션만 재시도하고 다른 세션은 다음 쿼리부터 새 핸들
  사용.
- 라우터 폴백 (`app.py`): route_query의 Gemini 호출을 try로 감싸
  레이트 리밋(429) 등 API 오류 시 앱 크래시 대신 ("rule_db", "lookup")
  폴백.
- 임베딩 직렬화 (`app.py`): 전 세션 공유 SentenceTransformer의 동시
  encode() 호출을 threading.Lock으로 직렬화 (MPS 백엔드 동시 호출
  크래시 방지).
- Q&A 로그 원자화 (`app.py`): append_qa_log를 단일 write로 변경 —
  동시 기록 시 사용자 간 줄 교차 방지.
- 로그 세션 태그 (`core/logging_config.py`): contextvar 기반
  세션 필터 추가 — 모든 로그 줄에 세션 식별자 컬럼 자동 표기
  (`... | abc123 | aos.pipeline | ...`), 동시 질문 로그 구분 가능.
  앱은 질문 시작 시 set_log_session() 호출.

**검증**
로그 필터(미설정 "-" 폴백, 스레드별 격리) 확인, 같은 초 세션 ID 1000개
전부 유일, 클라이언트 캐시 갱신 후 쿼리 정상(기존 핸들과 교체 확인),
전체 컴파일 통과.

**미해결 과제**
Gemini 무료 쿼터는 여전히 공유 자원 — 사용자가 늘면 세션당 질문 수
제한 또는 유료 티어 검토 필요.

### 4. 무기 프로필 표준 표기 형식 지정

**내용** (`app.py`)
faction_db·spearhead_db 답변 프롬프트에 무기 프로필 표기 형식 지정:
`| 무기 | 사거리 | 공격횟수(Atk) | 명중(Hit) | 피해(Wnd) | 관통(Rnd) | 대미지(Dmg) | 특수 능력 |`
마크다운 표. range가 비어 있으면(근접 무기) '근접', 값이 있으면 인치

누락 0건. 스크래핑 데이터가 표기 형식 전제와 완전히 일치.

**검증**

### 5. 유닛 스탯 표준 표기 형식 지정 + 데이터 검증

**내용** (`app.py`)

'이동(Move)', '체력(Health)', '방호(Save)', '점령(Control)' (한글+영문
병기), ward 값이 있으면 방호 다음에 '와드(Ward)'를 넣어

  체력·점령(숫자) 형식 전부 정상. 와드는 div.wsWard에서 스크래핑돼
  423개 유닛이 3+~6+ 값 보유, 나머지는 빈 값 — 사용자 설명("wsWard에
  숫자가 있으면 와드")과 데이터 구조 일치.
- 스탯 빈 유닛 558개 발견: 매니페스테이션 59개(별도 프로필 구조) +
  Regiments of Renown 참조 블록 485개(스탯 없이 능력 텍스트만, 전
  팩션 페이지에 중복 게재). RoR 블록은 이전 검색 오염(Drekki's
  Privateers)의 원인이기도 함.

**검증**
와드 보유(Grombrindal 3+)/미보유(Aether-Khemist) 유닛 e2e — 병기 라벨,
와드 위치(방호 다음), 미보유 시 생략 모두 정확.

**미해결 과제**

### 6. 무기 능력 정의 자동 주입 (Companion, Crit 등)

**증상**
"Warsteed's Hooves의 Companion 설명해줘"에 4번 시도 끝에야 답변 —
무기 특수 능력의 정의는 코어 룰(20.0 Weapon Abilities)에만 있는데,
질문은 유닛/스피어헤드 문맥이라 라우팅이 항상 spearhead/faction_db로
가서 컨텍스트에 정의가 없음. 사용자가 "코어 룰북에서 찾아줘"라고
명시해야만 rule_db로 감.

**데이터 확인**
무기 능력 어휘는 기본 패턴 10여 종(Companion 333회, Shoot in Combat,
Crit 계열, Anti-X, Charge (+1 Damage), Blood-hungry)이고, 정의는
rule_db의 "20.0 Weapon Abilities" 청크 3개(코어 룰 2 + FAQ 1)에 집중.

**수정** (`app.py`)
용어 사전 기반 컨텍스트 주입(팩션 로스터 주입과 동일 패턴):
- `load_weapon_ability_chunks()`: 정의 청크 3개를 시작 시 1회 로드(캐시).
- `WEAPON_ABILITY_RE`: 용어 감지(영문 패턴 + 컴패니언/크릿/무기 능력).
- 검색 완료 후 질문 또는 검색된 문서에 용어가 보이면 라우팅 DB와
  무관하게 정의 청크를 컨텍스트에 추가(중복 방지, 결과 0건이어도 주입).
- 라우팅을 바꾸지 않으므로 워스크롤(어느 무기에 붙었는지)과 정의
  (효과)를 한 컨텍스트로 함께 제공.

**검증**
감지 regex 9케이스 통과(용어·문서 스캔·한글 표기 발동, healing/유닛
목록/일반 차지 질문 미발동). e2e — 실패했던 원 질문 재현: Darkoath
Raiders 확장 + 정의 주입 → Companion 효과("아군 수정 능력의 영향을
받지 않음, 탈것 강화 제한")를 정확히 설명.

### 7. 일반 대화(chat) 모드 추가

**증상**
"당신은 뭘 알려줄 수 있나요?", "초보자를 위한 팩션 추천해주세요" 같은
일반 대화 질문에 "찾을 수 없습니다"로만 응답. 라우터가 빈 other_db를
고르고, 키워드 추출은 프롬프트 템플릿 잔해('질문: ... 출력:')를 뱉는 등
검색 파이프라인 자체가 대화형 질문에 무의미.

**설계 결정**
분석 모드 재활용안은 기각 — 분석 모드도 문서 근거에 묶여 있어("관련
규칙이 없으면 분석 금지") 검색 0건이면 동일하게 실패. 대신 라우터의
세 번째 유형으로 chat을 추가(추가 API 호출 0).

**수정** (`app.py`)
- 라우터 규칙 0: 인사/자기소개·기능 문의/입문 조언/팩션 추천 등은
  "chat" 출력. 특정 규칙·유닛·포인트 질문은 절대 chat 금지 명시.
- chat 판정 시: 키워드 추출·벡터 검색·주입 전부 생략하고
  CHAT_SYSTEM_PROMPT로 바로 답변(thinking 1000). 기능 안내는 실제 앱
  기능 6종+예시 질문을 프롬프트에 명시, 추천 질문은 지원 팩션 28종
  목록을 근거로 답변. 규칙 수치·스탯을
  기억으로 답하는 것은 금지 유지.
- 안전장치 상호작용: 질문에 스피어헤드 고유명 등이 있어 db_name이
  교정되면 chat을 취소하고 검색 경로로 복귀(프로필 재산정).
- KNOWN_FACTIONS를 핸들러 지역 변수에서 모듈 레벨로 승격(chat 프롬프트
  재사용).
- 부수 수정: 규칙 0 추가로 "strike-last의 장점" 라우팅이 faction_db로
  흔들리는 회귀 발견 → 라우터 규칙 4에 "용어의 장단점·평가·활용법도
  rule_db" 보강 (3회 반복 rule_db|analysis 일관 확인).

**검증**
라우터 라이브 테스트 9케이스 — chat 4종(기능 문의/추천/인사/입문) 전부
chat 판정, 데이터 질문 5종 기존 라우팅 유지. e2e — 기능 문의는 실제
기능 6종을 예시와 함께 안내, 팩션 추천은 실존 팩션(스톰캐스트,
아이언조즈 등) 기반 입문 조언 생성.

### 8. 조기 응답 후 스피너가 화면에 남는 UI 버그

**증상**
chat 모드 답변(팩션 추천 등)이 완료된 뒤에도 "지그마의 서고를 뒤적이는
중..." 스피너가 계속 표시됨.

**원인**
chat 분기와 배틀팩 반문 분기가 `st.spinner()` 컨텍스트 안에서
`st.stop()`을 호출 — Streamlit에서 스피너 안에서 스크립트를 중단하면
스피너 정리가 화면에 반영되지 않고 남는 문제.

**수정** (`app.py`)
파이프라인을 3구간으로 재배치:
1. "질문을 읽는 중..." 스피너 — 재작성·라우팅·안전장치·반문 판정
   (판정은 flag로만, 렌더링 없음)
2. 스피너 밖 — 조기 응답 분기: 배틀팩 반문 렌더링/chat 답변
   (생성만 "답변을 작성하는 중..." 스피너로 감싸고 렌더링과 st.stop()은
   스피너 밖에서)
3. "지그마의 서고를 뒤적이는 중..." 스피너 — 검색·답변 생성 (기존 경로)

### 1. 룰 용어 질문 검색 실패 ("healing" 미검색)

**증상**
"코어룰에서 healing이라는 용어를 찾아서 알려줘" 질문에
"제공된 문서에서 해당 정보를 찾을 수 없습니다"로 응답.
원본 데이터와 벡터 DB에는 "21.0 Healing" 청크가 정상 존재했음.

**원인**
`generate_search_query`의 키워드 추출 프롬프트가 팩션/유닛/스피어헤드
이름 추출 규칙만 갖고 있어, 룰 용어 질문에서 flash-lite가 유일한
고유명사처럼 보이는 '코어룰'을 잡아 `'CORE RULES'`로 추출하고 정작
검색 대상인 `healing`을 버림. 'CORE RULES' 임베딩 검색은 배틀팩의
"THE CORE RULES" 안내 청크 등 개요 문서만 상위 10개를 채움.

**수정** (`app.py`)
- 추출 프롬프트에 규칙 5 추가: 룰 용어 질문은 그 용어의 영어 키워드만
  출력, '코어룰'·'core rules' 같은 문서 이름은 검색어에서 제외.
  예시 3건 추가 (healing → Healing, ward 세이브 → Ward, 렌드 → Rend).
- rule_db 병행 검색 도입: 추출 키워드 임베딩 + 재작성 원문 질문
  임베딩으로 각각 검색 후 병합. 추출 LLM이 키워드를 훼손해도 원문
  검색이 안전망 역할.
- 병합은 거리순이 아닌 **쿼리별 순위 교차(rank interleave)** 방식 사용.
  일반적 키워드 쿼리는 거리 스케일이 전반적으로 낮게 나와 거리순
  병합 시 상위권을 독식하는 문제를 검증 과정에서 확인했기 때문.

**검증**
- 추출 재테스트: 문제 질문 → `'Healing'`, 'ward 세이브가 뭐야' →
  `'Ward'`. 기존 팩션/유닛 케이스 회귀 없음.
- 최악 케이스(추출이 여전히 'CORE RULES'를 내놓는 경우)에도 병행 검색
  병합 결과 2위에 "21.0 Healing" 청크가 포함됨.

### 2. 스피어헤드 배틀 진행 질문에 잘못된 배틀플랜·타 팩션 능력 인용

**증상**
"sand and bone에서 grundstok trailblazers 플레이, 배틀 시작부터 라운드
끝까지 과정" 질문에 ① 스피어헤드에 없는 배틀플랜 선택(GHB의 'Seize
the Embers') 안내, ② Grundstok Trailblazers가 아닌 다른 능력
(sand-and-bone의 커맨드, 타 배틀팩 배틀 택틱)만 예시로 제시.

**원인**
- 라우터 규칙 2("진행 순서 질문은 rule_db")에 걸려 **rule_db로 라우팅**
  됐는데, rule_db에는 팩션 스피어헤드 데이터가 0건. 검색어 'Grundstok
  Trailblazers'로 아무것도 못 찾고 임베딩상 가까운 GHB 매치드 플레이
  배틀플랜과 타 배틀팩(city-of-ash, fire-and-jade) 청크가 채워짐.
- 추가 발견: spearhead_db의 "스피어헤드 통째로 확장" 로직이
  `source.startswith("spearhead_")` 조건인데 현재 소스명은 전부
  `wahapedia_*`라서 한 번도 작동한 적 없는 죽은 코드였음.

**수정** (`app.py`)
- 라우터 프롬프트에 예외 추가: 특정 배틀팩/스피어헤드 이름이 언급되면
  spearhead_db. 코드 레벨 안전장치도 추가(배틀팩 이름 regex
  `BATTLEPACK_SOURCES` 감지 시 rule_db → spearhead_db 강제 교정).
- 병행 검색을 spearhead_db에도 적용(배틀팩+유닛처럼 대상이 둘인 질문 대응).
- 배틀팩 특정 시 다른 배틀팩 소스를 `$nin` 필터로 제외(오염 차단).
- 확장 로직을 `spearhead_name` 메타데이터 기반으로 재작성:
  질문에 실제 언급된 스피어헤드(오타 교정된 search_query로 대조)만
  전체 확장(배틀 트레잇·레지먼트 어빌리티·인핸스먼트·워스크롤),
  임베딩 노이즈로 걸린 무관한 스피어헤드 청크는 제거,
  배틀팩 특정 시 해당 배틀팩 문서 전체를 확장해 시퀀스 누락 방지.
- spearhead_db 답변 프롬프트에 지침 추가: 능력 예시는 질문한
  스피어헤드 것만 사용, 스피어헤드 모드에는 배틀플랜 선택이 없음.

**검증**
동일 질문 재현 시 라우터가 spearhead_db 선택, 최종 컨텍스트가
Grundstok Trailblazers 전체 7청크 + Sand and Bone 배틀팩 전체
36청크로만 구성됨(GHB·타 배틀팩·타 팩션 오염 0건, 총 약 38KB).
"팩션 스피어헤드 목록" 질문도 회귀 없음.

### 4. DB별 유사도 조사가 빈 컬렉션에서 중단되는 문제

**증상**
`[4.검색] DB별 유사도 조사 실패` 경고와 함께
`chromadb.errors.InternalError: ... hnsw segment reader: Nothing found
on disk` 트레이스백. 본 검색과 답변은 정상.

**원인**
`other_db`(특수 캠페인 DB)가 0건이라 HNSW 인덱스 파일이 디스크에
없음. 진단용 유사도 조사가 5개 컬렉션을 순회하다 여기서 실패했고,
try가 루프 전체를 감싸고 있어 조사 전체가 중단됨.

**수정**
- `app.py`: 빈 컬렉션(`count()==0`)은 건너뛰고, 컬렉션별 개별 try로
  변경. 실패 시 `유사도 조사 실패한 DB: other_db(InternalError)`
  형태로 한 줄만 기록.
- `core/logging_config.py`: 서드파티 로거(httpx, httpcore,
  google_genai)를 WARNING 레벨로 올려 파이프라인 로그 가독성 개선.

**검증**
빈 other_db 포함 실제 컬렉션으로 실행 — 에러 없이 4개 DB 조사 완료,
'Hurakan Vanguard' 최근접 DB(spearhead_db)가 라우터 판정과 일치 확인.

**미해결 과제**
`other_db`가 0건인 상태. 특수 캠페인(기란의 재앙 등) 질문이 라우팅되면
아무것도 찾지 못함. 캠페인 데이터를 쓸 계획이면 `build_db.py`의
other_db 적재 누락 여부 확인 필요.

### 5. 앱 실행 중 Chroma 핸들 무효화로 본 검색 크래시

**증상**
사격 페이즈 질문 처리 중 본 검색(`collection.query`)에서
`chromadb.errors.InternalError: ... hnsw segment reader: Nothing found
on disk`로 앱이 크래시. 새 프로세스에서 같은 쿼리는 5개 컬렉션 모두 정상.

**원인**
DB 파일은 정상. 실행 중인 Streamlit 앱이 `st.cache_resource`로 캐시한
Chroma 컬렉션 핸들이, 외부 프로세스의 DB 재빌드/컴팩션 이후 사라진
세그먼트 파일을 가리키는 상태. chromadb는 프로세스 내에서 같은 경로의
클라이언트를 재사용하므로 캐시만 비워서는 복구되지 않음.
(항목 4의 유사도 조사 실패도 동일 메커니즘이었을 가능성 높음)

**수정** (`app.py`)
본 검색을 `chromadb.errors.ChromaError`로 감싸고, 실패 시
`st.cache_resource.clear()` + `SharedSystemClient.clear_system_cache()`
로 두 캐시를 모두 비운 뒤 리소스를 재로딩해 1회 재시도.

**검증**
클라이언트 캐시 초기화 → 재생성 → 쿼리 시퀀스가 동일 프로세스에서
정상 동작함을 확인. 기존 앱 프로세스는 재시작 필요(1회).

### 7. 대화 문맥 재작성 실패 보정 ("각 유닛" 등 지시 표현)

**증상**
"Grundstok Trailblazers에는 어떤 유닛이 있어?" 다음 턴의 "각 유닛의
워스크롤을 알려줘"가 재작성되지 않고([1.재작성] 변경 없음) 키워드
추출이 'WARSCROLL'이 되어 무관한 문서(엔들리스 스펠)가 검색됨.

**원인**
같은 입력을 다시 돌리면 재작성이 성공하는 경계 사례 — 프롬프트의
보수 규칙("완결이면 그대로", "언급 안 된 이름 추가 금지")이 강해
temperature=0에서도 비결정적으로 성공/실패가 뒤집힘. 부차 요인:
① 이력 200자 절단으로 유닛 목록 유실, 출처 푸터가 절단 예산 잠식,
② 재작성 실패 시 받쳐줄 안전망 부재.

**수정** (`app.py`)
- ① 이력 전달 개선: 어시스턴트 메시지의 출처 푸터(--- 이후) 제거 후
  400자로 확대, 최근 6개(3턴) 전달.
- ② 프롬프트 보강: 지시 표현 예시에 집합 지시어("각 유닛", "전부",
  "나머지" 등) 추가, "직전 답변의 목록을 가리키면 목록의 주체를 붙여
  재작성" 규칙과 Grundstok 예시 추가.
- ③ 결정적 안전망: 매 턴 search_query를 세션에 저장(last_search_query).
  재작성 결과가 원문과 동일한데 원문에 지시 표현(DEMONSTRATIVE_RE:
  각/그/이/해당/전부/모두/나머지 등)이 있고 직전 주제가 질문에 없으면
  "{직전 주제} — {질문}"으로 결합. LLM이 흔들려도 코드 레벨에서 보정.

**검증**
실제 실패 세션 데이터로 3회 반복 — 전부 "Grundstok Trailblazers의 각
유닛의 워스크롤을 알려줘"로 일관 재작성. 새 주제 질문(sand and bone
규칙)은 그대로 유지(오발동 없음). 안전망 regex 6케이스 통과("유닛이
이동할..." 같은 조사 '이'는 미발동).

### 8. 배틀팩 미지정 스피어헤드 진행 질문에 원론적 답변만 나오는 문제

**증상**
"일반 스피어헤드 모드의 게임 진행은 어떻게 하는 거야?"에 제품 소개
수준의 원론적 답변("가장 간단한 방법", "높은 재플레이성" 등)만 나오고
실제 진행 규칙은 "찾을 수 없습니다"로 끝남.

**원인**
파이프라인 로그로 확인 — 재작성·라우팅은 정상이었으나, 'General
Spearhead Mode' 같은 일반 키워드로는 각 배틀팩 파일의 "SPEARHEAD"
소개(홍보문) 청크만 상위 검색되고 실제 진행 규칙 청크("1. PICK A
SPEARHEAD ARMY", "5. FIGHT THE BATTLE!" 등)는 컨텍스트에 못 들어옴.
근본적으로 진행 규칙(트위스트/배틀 택틱/배틀플랜)은 배틀팩(시즌)마다
달라, 배틀팩 미지정 진행 질문은 검색으로 답할 수 없는 질문임.

**수정** (`app.py`)
- 결정적 반문: 질문(재작성 후)에 스피어헤드 + 진행 표현(진행/게임
  방법/어떻게 플레이/순서/시작부터)이 있는데 배틀팩이 미지정이면,
  검색·LLM을 생략하고 배틀팩 4종(Fire and Jade / Sand and Bone /
  City of Ash / Spearhead Doubles) 목록과 질문 예시를 제시하는 반문
  응답을 바로 출력 (`CLARIFY_BATTLEPACK_MSG`, `SPEARHEAD_PROGRESSION_RE`).
- rule_db 답변 프롬프트 보강: 반문 감지를 비껴간 일반 스피어헤드 규칙
  질문도, 검색 문서가 소개 수준뿐이면 개요를 늘어놓지 말고 배틀팩을
  되묻도록 지침 추가.
- 배틀팩을 지정해 다시 질문하면 기존 로직(spearhead_db 교정 + 배틀팩
  전체 확장)이 진행 규칙 전체를 컨텍스트로 공급함 (항목 2에서 구축).

**검증**
트리거 10케이스 통과 — 실제 실패 질문·변형(게임 방법/플레이/순서)은
반문 발동, 배틀팩 지정 질문("sand and bone에서 게임 진행")·특정 룰
질문(차지 규칙)·유닛 질문·일반 코어 룰 진행 질문은 미발동.

### 9. 분석/의견 모드 추가 (규칙 해석·평가형 질문 대응)

**증상**
"strike-last의 장점이 뭐야?" 같은 평가·해석형 질문에 "제공된 문서에서
찾을 수 없습니다"로만 응답. 규칙 원문에는 '장점'이 명시돼 있지 않으므로
당연한 결과 — 답변 프롬프트의 할루시네이션 방지 규칙("유추·비유 금지,
없으면 못 찾음이라고만")이 해석형 질문까지 차단하는 구조적 한계.

**수정** (`app.py`)
- `ANALYSIS_RE`: 분석형 질문 감지(장점/단점/평가/전략/활용/시너지/
  추천/비교/차이/상성/카운터/어때 등).
- 감지 시 답변 시스템 프롬프트에 `ANALYSIS_ADDENDUM` 부착: ① 관련 규칙
  원문을 근거로 먼저 요약 → ② '분석 (의견)' 소제목 아래 규칙 근거를
  들어 게임적 의미·장단점 해석 → ③ 의견임을 답변 끝에 명시 →
  ④ 존재하지 않는 규칙·수치를 지어내는 것은 여전히 금지, 관련 규칙
  자체가 없으면 분석도 금지.
- 발동 시 로그: `[7.답변] 분석 모드 활성화`.

**검증**
- 감지 regex 8케이스 통과(분석형 4종 발동, 조회형 4종 미발동).
- e2e: 실제 실패 질문으로 검색+답변 재현 — STRIKE-LAST 규칙 요약(출처
  번호 표기) 후 '분석 (의견)' 섹션에서 "자체는 페널티이며, 강력한
  이점과 교환으로 받아들이는 경우(City of Ash의 Reinforcements 택틱
  등)의 간접적 장점" 형태의 근거 기반 해석 생성 확인.

### 10. 조회형/분석형 답변 프로필 이원화

**변경** (`app.py`)
항목 9의 분석 모드를 프로필 체계로 승격:
- `ANSWER_PROFILES`: 모드 → (모델, DB별 thinking 예산) 매핑.
  - lookup: gemini-2.5-flash, thinking 낮춤 (rule_db 8000→4000 등) —
    조회는 발췌·정리 작업이라 과한 추론 예산이 불필요.
  - analysis: gemini-2.5-flash, thinking 대폭 상향 (rule_db 16000 등).
    품질 부족 시 model 한 줄만 gemini-2.5-pro로 교체하면 됨.
- 라우터 승격: ROUTER_PROMPT가 DB와 질문 유형을 함께 분류
  ("rule_db|analysis" 형식). route_query는 (db_name, query_type) 반환.
  추가 API 호출 없음. ANALYSIS_RE는 안전망으로 유지(둘 중 하나라도
  분석 판정이면 분석 모드).
- 로그: `[2.라우터] 판정: rule_db / analysis (라우터=…, regex=…)`,
  `[7.답변] 프로필: analysis (model=…, thinking=…)`.

**검증**
실제 라우터 호출 6케이스 — 유형 분류 6/6 정확(장점/쓸만해/활용 →
analysis, 용어/유닛 목록/포인트 → lookup), "형식(db|type)" 파싱 정상,
DB 라우팅은 기존 동작과 동일(회귀 없음).

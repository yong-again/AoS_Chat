## 1. RAG 파이프라인 오프라인 평가 하네스 추가 + 베이스라인 측정

**증상(배경)**: 파이프라인 수정이 누적되면서 회귀 여부를 사람이 채팅으로 일일이
확인해야 했고, 정량적인 성능 기준선이 없었다.

**수정**: `data/eval/rag_eval_qa.json`(19문항: faction 6, rule 4, balance 4,
spearhead 3, other 2 — 각 문항에 기대 답변·근거 문서·기대 라우팅·필수 키워드 포함)
기반의 평가 하네스 `scripts/eval_rag.py`를 추가.

- app.py를 bare 모드로 import해 **실제 파이프라인 코드**(라우팅 → 4중 안전장치 →
  키워드 추출 → 듀얼 임베딩 + BM25/RRF 하이브리드 → 보강/폴백 → 답변 생성)를
  단일 턴으로 재현 실행 (사이드바 기본값 조건: 패치 포함, 과거 GHB 제외)
- 측정 지표 4종: ① 라우팅 정확도 ② 검색 적중률(영문 키워드가 컨텍스트에 포함)
  ③ 답변 키워드 커버리지 ④ LLM 심사(flash-lite, 기대 답변 대비
  correct/partial/incorrect)
- 레이트 리밋 대비 클라이언트 레벨 재시도(5→60초 백오프), 문항별 중간 저장
- 실행: `python -m scripts.eval_rag [--only 1,5] [--no-judge]`
  → 결과: `data/eval/results/eval_<ts>.json`

**검증(베이스라인, eval_20260710_115309)**:

## 2. 평가로 발견된 신규 버그: 검색 0건일 때 폴백 문서가 컨텍스트에서 유실

**증상**: Q19(other_db 라우팅, 컬렉션 비어 있음)에서 프로브 크로스 DB 폴백이
faction_db 문서를 추가했는데도 최종 컨텍스트가 "관련 문서를 찾을 수 없습니다"로
조립되어 폴백이 무효화됨.

**원인**: app.py의 폴백들(키워드 폴백, 스피어헤드 크로스, 프로브 크로스 DB)은
`results["documents"][0]`/`metadatas`만 앞에 붙이고 `ids`는 갱신하지 않는다.
1차 벡터 검색이 1건이라도 있으면 컨텍스트 조립 루프가 documents를 돌아 문제가
없지만, **0건이면** 조립 가드(`if results["ids"] and len(results["ids"][0]) > 0`)가
False가 되어 폴백으로 추가된 문서 전체가 버려진다.

**수정**: 세 폴백(키워드 폴백, 스피어헤드 크로스, 프로브 크로스 DB) 모두에서
documents/metadatas와 함께 `results["ids"][0]`도 갱신하도록 수정 (app.py).

**검증**: 재현 조건이었던 Q19는 아래 3번(other_db 적재) 이후 1차 검색이 문서를
찾아 해당 경로를 더 타지 않음 — 코드 레벨 수정으로, 세 폴백 지점 모두 ids를
같은 순서로 prepend함을 확인.

## 3. other_db 적재 — Scourge of Ghyran 캠페인 질문 답변 불가 해소 (Q18/Q19)

**증상**: other_db가 0건이라 Scourge of Ghyran 질문(Q18/Q19)에 답변 불가.

**원인**: build_db.py가 PDF 추출본(outputs/)을 `--with-pdf`일 때만 적재하는데,
other_db는 wahapedia 소스가 없어 PDF 추출본이 유일한 소스 — 기본 실행에서 항상
비게 됨.

**수정** (build_db.py):

- other_db는 `--with-pdf` 없이도 항상 `data/outputs/other_db/`에서 적재
- `--only <컬렉션>` 옵션 추가: 지정 컬렉션만 삭제·재적재하고 나머지는 유지
  (전체 재구성 없이 `python build_db.py --only other_db`로 부분 재구성)

추가로 검색 품질 보정 (app.py):

- 라우터 프롬프트의 other_db 설명 구체화(캠페인 이름이 팩션과 함께 나와도
  other_db) + 안전장치 5: `SCOURGE_RE`(영문/음차/번역 표기) 감지 시 other_db 교정
- other_db N_RESULTS 5→25 (캠페인 청크는 캠페인 이름이 전 청크 공통이라 변별력이
  낮아 RRF 컷에서 밀리지 않도록)
- 원문 질문 병행 임베딩 검색 대상에 other_db 추가(키워드 추출이 캠페인 이름만
  남기고 주제를 떨어뜨리는 경우 대비) + 추출 프롬프트에 캠페인 질문 예시 추가

**검증**: `--only other_db` 재구성 후 212청크 적재. Q19 correct,
Q18 incorrect→partial(능력 이름·조건·효과 정답, 전투당 1회 타이밍만 누락).

## 4. 스피어헤드 이름 한국어 음차 매칭 실패 수정 (Q16)

**증상**: "놀피스트 클로팩"(= Gnawfeast Clawpack)처럼 한국어 음차로 쓴
스피어헤드 이름이 영문 고유명 사전과 매칭되지 않아 안전장치 미발동 →
faction_db 오라우팅, 다른 스피어헤드의 인핸스먼트로 오답.

**수정** (app.py):

- 키워드 추출 프롬프트에 규칙 9 추가: 음차 고유명을 공식 영어 철자로 변환하되,
  스피어헤드 이름은 프롬프트에 주입된 공식 명칭 목록(`load_spearhead_names()`)
  에서 골라 그대로 출력
- 안전장치 6(추출 후) 추가: 추출된 검색어에 스피어헤드 고유명이 있으면
  spearhead_db로 교정 — 원문이 음차라 안전장치 3(영문 사전 매칭)이 빗나가도
  추출 단계의 영문 변환 결과로 결정적으로 잡는다

**검증**: Q16 incorrect→correct (Gnawfeast Clawpack 인핸스먼트 정답).

## 5. 유닛 능력 질문의 "규칙" 단어로 인한 rule_db 오라우팅 보정 (Q5)

**증상**: "레이디 올린더 때리려는데 공격이 잘 안 박히는 규칙이 있다던데?" —
"규칙" 단어에 끌려 rule_db 라우팅(기대 faction_db) + 키워드 추출이 고유명 오타
생성(Olynder→"Olonder")으로 검색 실패.

**수정** (app.py): 라우터 프롬프트 규칙 4에 "'규칙' 단어가 있어도 특정 유닛의
고유 능력이면 faction_db" 명시(올린더 예시 포함). 추출 프롬프트 규칙 9(음차→
공식 철자, 임의 발음 표기 금지)와 Lady Olynder 예시로 오타 생성 차단.

**검증**: Q5 라우팅 faction_db 정상화, incorrect→partial(MORTARCH OF GRIEF 핵심
설명 정답, Ward 5+ 부가 정보만 누락).
